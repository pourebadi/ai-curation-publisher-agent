# Telegram topic workflow MVP delivery notes

This zip is prepared as a Telegram-first MVP for fast manual testing. No GitHub repository was modified.

## What is completed for MVP testing

1. The core routing model is preserved: one source topic per category, multiple language outputs, independent review topic per language, independent final channel per language.
2. Output-level callback behavior is preserved: Send, Schedule, Cancel, and Status operate on the exact `generatedOutputId`.
3. Queue/schedule/final publishing remains independent per route output, language, and final channel.
4. D1 `media_processing_jobs` support is included for external media processing state.
5. GitHub Actions media processor workflow is included at `.github/workflows/media-processor.yml`.
6. `scripts/media_processor.py` uses `yt-dlp`, `ffmpeg`, compression/recompression, thumbnail generation, Telegram staging upload, and Worker callback.
7. External media processing can return up to 10 prepared Telegram `file_id` assets for valid album/media-group publishing.
8. Text-only social links do not block the MVP by default. Missing downloadable external media is marked `skipped`; set `MEDIA_PROCESSING_STRICT=true` when media is mandatory.
9. The Worker dispatches GitHub media jobs only for likely media/social/video URLs, not every article URL.
10. Direct Telegram media uploaded in source topics still uses Telegram `file_id` reuse and does not need GitHub Actions.
11. Publish runner waits while media jobs are pending, fails only the affected output if a strict media job fails, and keeps queue rows isolated by language/channel.
12. Dashboard includes an MVP route builder form for one source topic with multiple language outputs.
13. Dashboard Activity shows generated outputs, publish queue rows, and media jobs, with a guarded due-publish trigger.
14. `scripts/telegram-mvp-smoke.mjs` can seed a route, simulate source ingest, optionally simulate Send, and optionally trigger due publish.
15. `scripts/telegram-set-webhook.mjs` helps set the Telegram webhook for the deployed Worker.
16. WordPress remains optional for this Telegram topic workflow.

## Safe defaults

```text
TELEGRAM_FINAL_PUBLISH_ENABLED=false
TELEGRAM_PUBLISH_SCHEDULER_ENABLED=false
GITHUB_MEDIA_PROCESSOR_ENABLED=false
MEDIA_PROCESSING_STRICT=false
```

Enable final publishing and scheduler only after review messages and queue rows are verified.

## Media processor boundary

The GitHub Actions media processor is an MVP implementation, not a guarantee that every X/Instagram URL will download. It works best with public direct media URLs and public yt-dlp-supported links. Private, login-gated, age-gated, or blocked media may require `X_COOKIES_B64` or `INSTAGRAM_COOKIES_B64`. It does not bypass platform restrictions.

## Reference repo influence

The `social-review-curator-agent` repository uses a media-first pattern with X/Instagram extraction, yt-dlp, ffmpeg, local video thumbnails, Telegram staging/publishing, and manual approval before publishing. This MVP borrows that operational shape but fits it into the current Cloudflare Worker + D1 + topic-routing architecture.
